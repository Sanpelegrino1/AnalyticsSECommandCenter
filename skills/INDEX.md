# Skill Index

Use this file as the first stop before running repeated operational work.

| Skill | When to use it | Required inputs | Validation |
| --- | --- | --- | --- |
| `salesforce-org-login-and-alias-registration.md` | New org login or alias refresh | Alias, login URL, purpose, notes | Alias appears in `sf org list` and registry is updated |
| `switch-active-salesforce-org-context.md` | Change active Salesforce org | Alias | `sf config get target-org` or org open uses the expected alias |
| `run-soql-and-export-results.md` | Query org data for inspection or export | Alias, SOQL query, output path | Export file exists and record count looks correct |
| `retrieve-or-deploy-metadata.md` | Bring down or push Salesforce metadata | Alias, manifest path or metadata name | CLI command completes successfully |
| `upload-csv-to-data-cloud.md` | Bulk-upload CSV data into a configured Data Cloud ingestion target | Target key, CSV path, local auth secrets | Job reaches `JobComplete` or returns a diagnosable failure |
| `guided-data-cloud-manifest-setup.md` | Authorize Data Cloud and register multi-table manifest targets with minimal Salesforce navigation | Manifest path, login URL, alias, source name, naming prefixes | Token exchange succeeds and registry entries are generated |
| `provision-data-cloud-manifest-streams.md` | Register manifest schemas, create or reconcile Data Cloud streams, and confirm DLO activation before upload | Manifest path, Data Cloud alias, source name, object naming inputs | `provisioning-state.json` shows compatible schemas and `ACTIVE` DLO status |
| `publish-data-through-command-center.md` | Treat "I have data to publish" as the top-level user intent and orchestrate Data Cloud publication with optional Tableau Next follow-through | Dataset path, org alias/login URL if missing, Data Cloud alias, publication scope, optional naming and workspace inputs | Targets are registered, upload state is known, and optional SDM readiness or apply results are captured |
| `authenticated-to-sdm-orchestration.md` | Run the near zero-touch authenticated onboarding-to-SDM path and capture readiness state | Manifest path, Salesforce alias, Data Cloud alias, Tableau workspace selector or saved target, semantic-model naming inputs | Orchestration state file reports a readiness classification and, when requested, an applied semantic model plus manifest relationships |
| `create-lightning-app.md` | Build a Lightning app through a repeatable admin flow | Org alias, app name, visibility choices | App appears in Setup and App Launcher |
| `create-and-assign-permission-set.md` | Create access scaffolding for demo users | Org alias, permission set name, assignee | Permission set exists and user assignment succeeds |
| `open-key-salesforce-setup-surfaces.md` | Jump to common Setup pages quickly | Org alias, target surface | Browser opens the right Setup page |
| `tableau-cloud-auth-bootstrap.md` | Register or validate Tableau target auth | Target key, server URL, site content URL, PAT in local env | Auth status script succeeds |
| `list-tableau-cloud-projects-and-content.md` | Inspect Tableau project structure and content inventory | Target key | Project or content list returns without auth errors |
| `publish-or-inspect-tableau-content.md` | Publish workbook or datasource, or inspect content details | Target key, file path or content type and id | Publish returns success or inspect returns expected object |
| `author-tableau-next-dashboards.md` | Build Tableau Next visualizations and dashboards from an existing semantic model using the external authoring toolkit | Salesforce alias, workspace, semantic model, dashboard brief or spec path | Dashboard or visualization creation returns live ids or URLs |
| `author-tableau-next-semantic-layer.md` | Enrich an existing Tableau Next semantic model with calculated fields and semantic metrics | Salesforce alias, semantic model name, expression, field or metric metadata | Discovery output shows the created `_clc` or `_mtc` item |
| `package-and-deploy-tableau-next-content.md` | Package a Tableau Next dashboard from one org and validate or deploy it to another | Source alias, target alias, dashboard API name, package path, deployment choices | Package is created and validation or deployment completes with a concrete result |
| `share-tableau-next-assets.md` | Share Tableau Next workspaces, dashboards, or visualizations through the record-access API | Salesforce alias, asset id, object type, principal id, access level | Share list or POST response shows the expected access |
| `write-tableau-hyper-api-python.md` | Generate or review Python code for Tableau Hyper API extract workflows | Hyper task, schema, data source, optional publish requirement | Code follows the standard Hyper API lifecycle and load pattern |
| `inspect-tableau-next-target.md` | Register and validate a stable Tableau Next workspace-first target before later semantic-model creation | Salesforce alias, target key, workspace id, optional semantic-model id | Inspect script confirms the workspace exists and any pinned semantic model still resolves |
| `inspect-next-semantic-model.md` | Inspect one live Tableau Next semantic-model inventory row directly from a saved target or explicit IDs | Salesforce alias or saved target, workspace id, semantic-model id or workspace asset id | Inspect script returns `SemanticModelValidated` or `SemanticModelMissing` with the current asset row |
| `build-tableau-next-semantic-model-request.md` | Build or apply a manifest-driven Tableau Next semantic-model request from a saved target | Target key, model API name or spec path, manifest-derived object mappings and relationships | Helper returns a normalized spec plus supported semantic-model request and can apply the model live |
| `create-tableau-next-business-preferences.md` | Create business preferences that improve Tableau Next Analytics Q&A behavior for an existing semantic model | Semantic model context, schema or manifest details, business jargon, preferred defaults | Output is a concise `#`-prefixed preference block grounded in the real model; direct API write should only be used if repo support is later verified |
